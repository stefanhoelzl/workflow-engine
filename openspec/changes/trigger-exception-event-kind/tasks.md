### 1a. Architectural realignment (re-routes emission through the executor)

After the foundation landed, review surfaced that the original wiring let `TriggerSource` implementations import a stamping helper directly, breaking the existing source/registry contract (`packages/runtime/src/triggers/source.ts:16`: "Backends never touch the Executor — they invoke `entry.fire(input)`"). Aligning failure dispatch with handler dispatch:

- [x] 1.3a Moved the stamping primitive from `packages/runtime/src/triggers/exception.ts` (deleted) to `packages/runtime/src/executor/exception.ts`. Kept the `assertTriggerExceptionKind` chokepoint. Module documentation says it is consumed only by `executor.fail`; trigger sources MUST NOT import it directly.
- [x] 1.4a Added `executor.fail(owner, repo, workflow, descriptor, params)` to the `Executor` interface and implementation. Skips `SandboxStore` and `runQueue` entirely; delegates stamping to `emitTriggerException`.
- [x] 1.4b Added `buildException` at `packages/runtime/src/triggers/build-exception.ts`. Identity bound at build time; `(params) => executor.fail(...)`.
- [x] 1.4c Extended `TriggerEntry<D>` with `readonly exception: (params: TriggerExceptionParams) => Promise<void>`. Updated the source.ts module comment to cover the new exception channel ("Backends never touch the Executor or the EventBus directly — they invoke `entry.fire(input)` for handler dispatch and `entry.exception(params)` for author-fixable pre-dispatch failures").
- [x] 1.4d Updated `WorkflowRegistry` to call `buildException` alongside `buildFire`; both closures bind to the same `(executor, owner, repo, workflow, descriptor)`.
- [x] 1.4e Removed `emitTriggerException?` field from `ImapTriggerSourceDeps`, removed the import of `TriggerExceptionEmitter`, and undid the `main.ts` wire-up. IMAP source's import graph relative to main is now back to original; PR2 will route failures via `srcEntry.entry.exception(...)`.
