## Why

The `correlationId` for root events is generated by `ContextFactory.httpTrigger()` and passed through to `eventFactory.create()`, but the factory is the only consumer — there is a 1:1 relationship between trigger invocations and `create()` calls. Moving generation into the factory simplifies the API and is consistent with how `create()` already generates `evt_`-prefixed event IDs internally.

## What Changes

- `eventFactory.create()` drops the `correlationId` parameter and generates `corr_<UUID>` internally
- `ContextFactory.httpTrigger()` no longer generates or passes a correlation ID
- The `EventFactory` interface signature changes from `create(type, payload, correlationId)` to `create(type, payload)` — **BREAKING** (internal API only, no external consumers)

## Capabilities

### New Capabilities

_(none)_

### Modified Capabilities

- `event-factory`: The `create` method no longer accepts an external `correlationId` parameter; it generates the correlation ID internally.

## Impact

- `event-factory.ts` — interface and implementation change
- `context/index.ts` — remove correlation ID generation, update `create()` call
- Unit tests for event factory and context — update call signatures and assertions
- No effect on `derive()`, `fork()`, scheduler, event store, dashboard, or any downstream consumers
- No effect on QueueStore interface or manifest format
- No sandbox boundary implications
