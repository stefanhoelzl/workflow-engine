## Why

A hostile workflow author can flip a successful run into a failed run by deferring a throw past handler return — `setTimeout(() => { throw new Error("boom"); }, 0)` causes the worker thread to die via Node's unhandled-rejection escalation, and the run rejects with `worker exited with code …` even though the handler resolved cleanly. The escape route is `Callable.invoke`'s rejection-as-control-flow contract: when a guest-fired Callable rejects, `pluginRequest`'s rethrow chains a new promise that gets discarded by the host plugin (e.g. timers' `fire()`), and Node escalates the unobserved rejection to worker termination. This corrupts audit signal (operators cannot distinguish guest-induced run failures from genuine engine bugs), drops in-flight events between the throw and the worker's restore boundary, and forces a cold-start on the next run that amplifies the author's CPU budget. F-2 (`fix(sandbox): close host/sandbox boundary opacity`) closed the host→guest leak via the `GuestSafeError` taxonomy and `sanitizeForGuest` closure rule; F-3 closes the symmetric guest→host escape with the same architectural shape — convert rejection-as-control-flow at the boundary into structured envelope data.

## What Changes

- **BREAKING (host plugin contract; not workflow-author-visible).** `Callable.invoke` no longer rejects when the guest throws. The signature changes from `(...args) => Promise<GuestValue>` to `(...args) => Promise<CallableResult>` where `CallableResult = { ok: true, value: GuestValue } | { ok: false, error: GuestThrownError }`. Guest-originated throws (the `awaitGuestResult` rejection path and the synchronous `JSException` from `callGuestFn`) flow into the `{ ok: false, error }` branch, preserving the curated `GuestThrownError` produced by F-2's bridge work (name, message, guest stack, structured own-properties). Engine-side errors (`CallableDisposedError`, `marshalArg` throws, vm-disposed mid-call) continue to reject — they remain the "fail loud" signal for engine bugs.
- New non-enumerable brand symbol `Symbol.for("@workflow-engine/sandbox#callableResult")` on each envelope object, used by `pluginRequest` to discriminate Callable envelopes from non-envelope return values without shape-sniffing.
- `pluginRequest` (`packages/sandbox/src/plugin.ts`) gains an envelope-aware resolve path: when the wrapped `fn()`'s return resolves to a branded envelope, the resolver emits `prefix.response` (ok branch) or `prefix.error` (error branch, error field passed through unchanged from envelope.error) and resolves the outer promise with the envelope itself. Crucially, the resolver **never rethrows** on envelope-error — preventing the F-3 escape at the architectural choke point. The non-envelope rejection branch (engine bugs) keeps the existing rethrow + serializeLifecycleError path. emitError is split into `emitErrorFromException` (non-envelope path) and `emitErrorFromEnvelope` (envelope path) for intent clarity.
- `__sdk.dispatchAction` (`packages/sdk/src/sdk-support/index.ts`, the second in-tree Callable consumer alongside timers) migrates from `await handler(input)` rejection-catch to envelope inspection: `if (!result.ok) throw result.error;`. The thrown `GuestThrownError` flows back through F-2's `sanitizeForGuest` closure rule unchanged, preserving the existing guest-visible action throw shape.
- New SECURITY.md §2 R-13 — "Guest→host boundary opacity (audit integrity)" — formalising the invariant. Pairs with R-12 (host→guest opacity) under the boundary-opacity framing. Rationale calls out audit forgery as the primary attacker gain, cold-start CPU amplification as secondary.
- New `system.error` / `system.response` close events on the timer's frame on guest throw (the audit shape was always intended for this case but was silently swallowed by worker death). Wire shape unchanged: the `error` field on envelope-path emissions carries `{ name, message, stack, ...ownProps }` — additive over today's `LifecycleError` shape used by non-envelope paths.
- **Out of scope (tracked follow-up).** `serializeLifecycleError` continues to drop `.name` and most structured own-properties for non-envelope error paths (host dispatcher throws — `FetchError`, `MailError`, `SqlError` — and engine bugs). The audit trail therefore carries strictly less information for those paths than what `sanitizeForGuest` propagates to the guest VM. F-3 does not close this asymmetry; it is documented as a known limitation in `design.md` and tracked as a follow-up that touches `trigger.ts:57`, host-call-action, and every host dispatcher.

### Modified Capabilities
- `sandbox`: new requirements covering the Callable envelope contract, the brand-symbol discriminator, and pluginRequest's auto-unwrap behaviour. Companion to the existing "Host/sandbox boundary opacity for thrown errors" requirement (R-12, F-2) on the symmetric host→guest axis.
- `sandbox-stdlib`: timers' `fire()` documented as relying on the new contract for fire-and-forget safety; SDK dispatchAction's migration to envelope inspection captured. The web-platform `system.exception` leaf path (reportError / `__reportErrorHost`) explicitly distinguished from F-3's `system.error` close shape — different audit semantics for "uncaught throw," not unified by design.

## Impact

- **Affected packages.** `@workflow-engine/sandbox` (Callable type, makeCallable invoke, pluginRequest auto-unwrap, brand symbol export), `@workflow-engine/sdk` (`__sdk.dispatchAction` envelope inspection), `@workflow-engine/sandbox-stdlib` (timers' `fire()` no source change — inherits via pluginRequest auto-unwrap; new regression test covering F-3 scenario).
- **Tests.** `packages/sandbox/src/callable-reentry.test.ts` and `packages/sandbox/src/bridge-install-descriptor.test.ts` migrate `await expect(callable()).rejects` assertions to envelope inspection. New regression test in `packages/sandbox-stdlib/src/timers/timers.test.ts` covering: guest schedules `setTimeout(() => { throw }, 0)`, run resolves OK with handler's value, sandbox emits one `system.error` close event under the timer's frame, sandbox can be reused for a subsequent `run()` (worker survival assertion).
- **Wire shape.** Bus events unchanged in topology. Envelope is a worker-internal object that pluginRequest unwraps before any event reaches the bus. Persistence layout, EventStore queries, dashboard rendering all unaffected. The `error` field on envelope-path `system.error` emissions carries additional fields (`name`, structured own-properties) over today's `LifecycleError` shape — additive for existing readers.
- **Public TS API.** `Callable` and new `CallableResult` types are exported from `@workflow-engine/sandbox`. The brand symbol is exported. Out-of-tree plugin authors who upgrade `@workflow-engine/sandbox` will see TS errors at any `await callable()` site that treats the result as a raw `GuestValue` — the type system enforces the contract change. No silent runtime change.
- **Workflow-author surface.** Zero change. Callable is host-plugin-internal; guests never see it directly. `setTimeout(() => { throw })` from workflow code now (a) does not kill the run, (b) emits a `system.error` row in the dashboard naming the timer's frame. No `demo.ts` update required.
- **SECURITY.md.** New R-13 paired with R-12. R-4 (cleanup ordering) gains a one-line cross-reference.
- **CLAUDE.md.** New upgrade note covering the Callable contract change for monorepo contributors and any future out-of-tree plugin authors.
