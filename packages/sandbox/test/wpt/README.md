# WPT Compliance Harness

Runs the WinterCG Minimum Common API-applicable subset of
[web-platform-tests](https://github.com/web-platform-tests/wpt) inside the
QuickJS sandbox. Reports pass / skip / fail at subtest granularity.

## Commands

```
pnpm test:wpt:refresh [--sha <commit>] [--strict]
  Regenerates vendor/ + manifest.json from upstream WPT.
  --sha <commit>: pin to a specific commit (default: latest main).
  --strict:       fail if spec-referenced subtests disappeared upstream.

pnpm test:wpt
  Runs the suite. Reads manifest + spec, routes each test to pass/skip,
  runs applicable tests in parallel.
  Env overrides:
    WPT_CONCURRENCY   default max(4, cpus × 2)
```

`pnpm test` does NOT include the WPT suite — it's separate to keep the
local unit-test loop fast.

## Layout

```
packages/sandbox/test/wpt/
├── spec.ts                   ← source of truth — edit this
├── vendor/                   ← regenerated by refresh; committed
│   ├── manifest.json
│   ├── resources/
│   ├── common/
│   └── …                     (only files needed for pass-classified tests)
├── harness/
│   ├── match.ts              specificity + glob pattern matcher
│   ├── limited-all.ts        concurrency-capped Promise.all
│   ├── preamble.ts           PREAMBLE / POST_HARNESS / ENTRY source strings
│   ├── composer.ts           assembles per-test source
│   └── runner.ts             runWpt() — sandbox + watchdog + dispose
├── vitest.config.ts
└── wpt.test.ts               top-level-await runner
```

## spec.ts conventions

Flat map of glob patterns → `{ expected: "pass" | "skip", reason? }`.
Most-specific pattern wins; `skip > pass` breaks ties at equal specificity.

Subtest-level overrides use `"file.any.js:subtest name"` (split at first colon).

```ts
"encoding/**":          { expected: "skip", reason: "first-run triage pending" },
"encoding/api-basics.any.js": { expected: "pass" },  // narrower override
"WebCryptoAPI/foo.any.js:specific subtest": {
  expected: "skip",
  reason: "needs wasm-ext XYZ feature",
},
```

Reasons that say `"needs X polyfill"` drive the polyfill backlog — grep the
file to see what's queued:

```
grep '"needs .* polyfill"' spec.ts
```

## Drift detection

Two checkpoints:

- **Refresh-time**: `pnpm test:wpt:refresh --strict` fails if any spec
  `file:subtest` key points to a file no longer in the vendor.
- **Runtime**: the runner synthesizes a failing `it()` for any declared
  `skippedSubtests` whose name was never observed in the actual run
  output (suggests an upstream rename).

## Manifest shape

```json
{
  "wptSha": "...",
  "vendoredAt": "...",
  "tests": {
    "encoding/api-basics.any.js": {
      "scripts": ["resources/testharness.js", "common/utils.js"]
    },
    "fetch/api/basic/stream-response.any.js": {
      "skip": { "reason": "contains {{host}}/{{ports}} network dependency" }
    }
  }
}
```

A manifest entry is exactly one of:
- **runnable** — has `scripts` (and optional `timeout: "long"`, `skippedSubtests`)
- **structural skip** — has `skip: { reason }`, not vendored

No subtest inventory is stored — subtest names are observed at runtime.

## Adding a polyfill

When a polyfill lands in `packages/sandbox/src/`:

1. Flip the relevant `spec.ts` entry from `skip` → `pass` (dir-level or
   file-level as appropriate).
2. Run `pnpm test:wpt:refresh` to pick up newly-runnable files into the vendor.
3. Run `pnpm test:wpt`. Add narrower skips for subtests still failing.
4. Commit the updated `spec.ts` + vendor diff.

## Security

The harness is **test-only infrastructure**. `__wptReport` is installed per-run
via `sandbox.run(..., { extraMethods: { __wptReport } })` and is never present
in production sandbox construction. See `/SECURITY.md §2` for the allowlist.
