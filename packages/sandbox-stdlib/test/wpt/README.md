# WPT Compliance Harness

Runs the WinterCG Minimum Common API-applicable subset of
[web-platform-tests](https://github.com/web-platform-tests/wpt) inside the
QuickJS sandbox. Reports pass / skip / fail at subtest granularity.

## Commands

```
pnpm test:wpt:refresh [--sha <commit>]
  Regenerates vendor/ + manifest.json from upstream WPT. Vendors every
  worker-applicable file (skip.ts is applied at runtime, not vendor time).
  --sha <commit>: pin to a specific commit (default: latest main).
  Fails if any literal-path skip.ts entry no longer exists upstream.

pnpm test:wpt
  Runs the suite. Reads manifest + skip.ts; runs every file not matched
  by skip.ts. Pass is implicit.
  Env overrides:
    WPT_CONCURRENCY   default max(1, cpus / 2)
```

`pnpm test` does NOT include the WPT suite вЂ” it's separate to keep the
local unit-test loop fast.

## skip.ts conventions

Flat `Record<string, string>` of glob patterns в†’ reason string. **Pass is
implicit**: any applicable file not matched by an entry runs and is
expected to pass. Lookup is exact-key first, then glob fallback (see
`harness/match.ts` вЂ” `findReason(skip, key)`).

This file replaced the earlier `spec.ts` / `Expectation` envelope design
after the WPT directory was relocated to `packages/sandbox-stdlib/test/wpt/`.
The old specificity-scoring function `findMostSpecific` is gone вЂ” a
simple "exact key, then first matching glob" lookup was enough in
practice.

**No override mechanism.** A glob skip swallows everything underneath; if
you need a narrower file to pass, expand the glob into per-file entries
first. Each test should match at most one skip entry.

Subtest-level overrides use `"file.any.js:subtest name"` (split at first
colon).

```ts
"streams/**":              "needs web-streams-polyfill",
"url/url-searchparams.any.js":
  "needs URL.searchParams getter in quickjs-wasi URL extension",
"WebCryptoAPI/foo.any.js:specific subtest": "needs wasm-ext XYZ feature",
```

Reasons that say `"needs X polyfill"` drive the polyfill backlog вЂ” grep
the file to see what's queued:

```
grep '"needs .* polyfill"' skip.ts
```

## Drift detection

Two checkpoints:

- **Refresh-time**: `pnpm test:wpt:refresh` fails if any literal-path
  entry in skip.ts no longer points to an applicable upstream file.
  Glob entries are not validated (they cannot point to a single file).
- **Runtime**: the runner synthesizes a failing `it()` for any declared
  subtest skip whose name was never observed in the actual run output
  (suggests an upstream rename).

## Manifest shape

```json
{
  "wptSha": "...",
  "vendoredAt": "...",
  "tests": {
    "encoding/api-basics.any.js": {
      "scripts": ["resources/testharness.js", "common/utils.js"]
    },
    "fetch/api/basic/stream-response.any.js": {
      "skip": { "reason": "contains {{host}}/{{ports}} network dependency" }
    }
  }
}
```

A manifest entry is exactly one of:
- **runnable** вЂ” has `scripts` (and optional `timeout: "long"`)
- **structural skip** вЂ” has `skip: { reason }`; only used for files with
  unresolvable network-dependent placeholders. Spec-level skips are NOT
  in the manifest вЂ” they live in `skip.ts` and are applied at runtime.

No subtest inventory is stored вЂ” subtest names are observed at runtime.

## Adding a polyfill

When a polyfill lands in `packages/sandbox-stdlib/src/`:

1. Remove (or narrow) the relevant `skip.ts` entry. If a glob skip
   covered a now-passing file, expand the glob into per-file skips for
   the still-failing siblings before removing the previously-passing
   file.
2. Run `pnpm test:wpt`. Add narrower skips for subtests still failing.
3. (Optional) `pnpm test:wpt:refresh` to pick up upstream changes.
4. Commit the updated `skip.ts` (+ vendor diff if you refreshed).

## Security

The harness is **test-only infrastructure**. `__wptReport` is installed via
the `WPT_HARNESS_PLUGIN` descriptor declared inline in `harness/runner.ts`
and only added to the sandbox's `plugins` array in the WPT test composition
вЂ” never in production runtime sandbox composition. The plugin registers
`__wptReport(name, status, message)` as a guest function whose `log.event`
is `"wpt.report"`; each guest call emits one event that `sb.onEvent` in the
runner accumulates into the subtest result list on the main thread. See
`/SECURITY.md В§2` for the top-level globals allowlist and the rules around
test-only sandbox surface.
